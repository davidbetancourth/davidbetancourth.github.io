Here lies equilibrium, among strategy and human greed.

Upsweep is a partial-information, zero-sum game designed from scratch for the Ethereum Blockchain.

The Upsweep Network

Welcome to the upsweep, a public and open network of 660 computer nodes arranged in a 2-dimensional graph. The objective of the game is to take over the network and mine as many sweeps as possible. However, the incentive goes beyond a sustainable gamble. Sweeps are an efficient store of value; free of speculation and always liquid.

Mining Mechanics

Mining sessions are scheduled by the upsweep clock and executed by the upsweep contract self-enforcing code that will always behave as expected. For each session, the contract will grant 60 players with remote access to the network. Before the competition begins, players will privately request a specific node (out of the 660 possible nodes) by sending the given (x,y) coordinate as a hash to the contract, after this, the competition will begin with a pre-committed and hidden distribution of 60 parent nodes. All players that went over the list will be included in a new list for the next session.

Account Limits

One single ethereum account can request up to 5 parent nodes for each session, but each request is a transaction on its own that must include a stake in ether that will serve as the weight of the node inside the network.
Players are only aware of their own node's position(s), and because every request is a secret and independent event, distributions are unpredictable.

Upsweep Private Clusters

Mining requires computational power, but one node is not enough. Players will attempt to form their own upsweep private clusters to fulfill the computational requirements. A minimum of 10 nodes must be allocated in order to participate in the mining race, but because players are only provided with a single parent node; they can request nearby and unused nodes to the network as soon as the competition begins.

All 660 nodes will appear as unused at the start of every competition, even pre-committed parent nodes. Players will decide if they want to activate their own parent node(s) or not, but to do so, they must select at least 9 more child nodes to meet the cluster requirement. Each of this child nodes will inherit their given parent node's data, and supply the cluster with identical computational power. Therefore, the total computational resources on a given cluster are equal to the weight of the parent node times the number of child nodes linked.
The order of events in a mining session is controlled by the upsweep clock and its internal protocol.

The Upsweep Clock

The upsweep network needs a unique timepiece to achieve equilibrium. The blockchain exists in discrete states called "blocks", and blocks happen to obey causality. On average, there is an interval of 15 seconds between blocks, during this period, players send transactions to the contract and influence the state of the next block. This time distortion between earth and the blockchain opens a channel for decision-making.
The upsweep clock is a key component of the contract. In a countdown fashion, it mimics the "holding" and "counting" scheme of NASA's countdown clock.
The upsweep clock repeats the following protocol in cycles, each cycle being an independent mining session:

E-30 blocks and holding

  • The contract is waiting for the full list of 60 parent nodes.

E-30 blocks and counting

  • Parent nodes have access to all limited resources of the upsweep network; they can request child nodes, activate clusters and prepare for mining execution.

E-0

  • No new clusters allowed
  • Existing clusters will execute mining operations according to their computational resources.
  • A new batch of sweeps will be created for every cluster.
  • Remote access expires for the current players and all node's data is erased for the next session.

Because parent nodes cannot mine without additional resources, players are in constant competition to form powerful clusters out of a limited set of 660 nodes. Creating a cluster is pretty easy, as soon as the upsweep clock hits the "E-30 blocks and counting" milestone, you can send transactions to the contract and request nodes. At E-0, no more requests are allowed and only existing clusters will be able to mine sweeps.
A formal request includes (x,y) coordinates for each child node that will be linked to the given parent node. If a valid request is received at block n, the cluster will be created at block n+1.

Rules for clusters

  1. Child nodes must connect to their parent node and all nodes must be connected by straight lines.
  1. A minimum of 10 nodes must be connected to form a cluster.
  1. A single parent node can request up to 25 nodes.
  1. If two or more players request the same node(s) at the same time (block), the parent node with the greater weight will be granted the given node(s). If parent nodes have equal weight, the node(s) will be granted to the request that results in a bigger cluster. If requests are equal, the node(s) will not be included in any of the clusters.

  1. If two or more players share the same parent node coordinate and they create a cluster at the same time (block), the cluster of the parent node with the greater weight will exist and the other(s) will not. If weights are equal, the request that results in a bigger cluster will exist and the other(s) will not. If requests are equal, none of the clusters will exist.
  1. Clusters can exist without their parent node, as long as they meet the minimum requirement of 10 nodes.

Overwriting Nodes

The hidden distribution of parent nodes creates panic. Players have two options, to either request and create their own cluster, or wait for other parent nodes to create a cluster that includes the node that they have remote access to. This is possible because parent nodes that are not part of a cluster will appear as unused and can be requested by other parent nodes.
If a parent node creates a cluster and includes your node, you can overwrite that child node's data and join the cluster as a sibling node. This type of node plays a key role in the game. A sibling node will supply the cluster with a computational power equal to its weight.

Cluster Reward

Mining gains depend on the computational power of the cluster. The reward of sweeps equals the weight of the parent node times the number of child nodes plus the weight of sibling nodes (if any).

sweeps_reward = (parentnode.weight * childnodes) + siblingnodes.weight

If there is sibling nodes in the cluster, parent and sibling nodes will share the reward according to their weight.

total_weight = parentnode.weight + siblingnodes.weight
parentnode_reward = (parentnode.weight / total_weight) * sweeps_reward
siblingnode_reward = (siblingnode.weight / total_weight) * sweeps_reward

Every player is better off alone

A cluster with sibling nodes might not be the best outcome but is not the worst. On the other hand, players that don't create clusters and are not included in any clusters will lose their stake.

Fear of Exclusion

The worst case scenario is to lose your stake, all of it. But this is only possible if you don't create a cluster or if you are not included as a sibling node in any other cluster. The later is not up to you and unfortunately, not being able to create a cluster is very plausible.

Invalid Cluster

If your cluster overlaps with another cluster and you lose nodes, there is a chance of not meeting the cluster requirements and losing your stake.

Claustrophobia

If you wait for other players to take initiative, you might end up in an uncomfortable position where there are not enough nodes nearby to meet the cluster requirement of 10 nodes.

Maximum Weight Rule

Upsweep is a game of timing. Players will rationalize that their best strategy is to race and create big clusters before nodes run out, but if everybody is rushing, overlaps will most certainly happen and only those with higher stakes will be less affected. Therefore, to mitigate the reach of this advantage, there is a maximum weight rule that limits the stake per parent node to 1 ETH. This will prevent manipulation and will also let players measure their odds with more precision.

Always Liquid

All the ether that goes into the game in the form of stakes is self-managed by the contract and no human has authority over it. No leaks and no fees, all ether goes straight to the market cap and the contract has explicit rules to always liquidate sweeps at the current market price. Liquidated sweeps are automatically burned to keep the price stable.
The upsweep is a smart contract that can't be shut down and will always exist for as long as the ethereum blockchain does. No one owns the upsweep, and no one controls it.
